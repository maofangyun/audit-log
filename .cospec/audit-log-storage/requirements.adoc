== 需求分析

=== 当前方案现状

当前系统采用 `sg-audit-starter` 审计日志组件，通过 AOP 切面注解方式和filter方式拦截请求并生成审计日志。
日志的持久化流程如下：

. 业务服务通过 `@AuditLog` 注解标记需要审计的方法
. `sg-audit-starter` 的切面拦截请求，采集操作信息（操作类型、资源类型、资源ID、变更前后状态等）
. **通过远程调用（WebClient）请求 platform 服务的保存接口**
. platform 服务接收日志数据，写入 PG 数据库

=== 当前方案痛点

==== 痛点一：强耦合 — 业务服务与 platform 服务的同步依赖

业务服务每次产生审计日志都需要同步调用 platform 服务的 HTTP 接口。当 platform 服务出现故障或响应缓慢时，
会直接影响业务服务的接口性能，甚至导致业务请求超时失败。审计日志本应是"旁路逻辑"，不应影响主业务流程。

==== 痛点二：性能瓶颈 — platform 服务成为系统单点

所有微服务实例的审计日志都汇聚到单一的 platform 服务进行持久化。在高并发场景下：

* platform 服务承受大量写入请求，CPU 和数据库连接池容易成为瓶颈
* 单点写入 PG 数据库，数据库连接数和写入吞吐量受限
* 随着日志量增长（10万-100万条/天），platform 服务的水平扩展能力有限

==== 痛点三：可靠性不足 — 日志丢失风险

* 远程调用受网络波动影响，调用失败则审计日志直接丢失
* 虽然可以加入重试机制，但重试会增加请求延迟，且无法无限重试
* platform 服务宕机期间产生的审计日志无法补录

==== 痛点四：资源浪费 — 审计日志占用业务线程

审计日志的序列化、网络传输、等待响应等操作都在业务线程中执行（即使异步发送也占用线程池资源），
在高并发场景下占用大量业务线程，影响系统的整体吞吐量。

=== 核心诉求

综合以上痛点分析，核心诉求如下：

[arabic]
. **解耦**：审计日志的生成与持久化必须完全解耦，业务服务不应感知日志的存储方式，日志写入失败不能影响业务流程
. **高性能**：日志采集过程对业务服务的性能影响降至最低，业务线程不参与日志的持久化操作
. **高可靠**：日志数据不能因任何单点故障而丢失，需要具备缓冲和重试能力
. **可扩展**：支持微服务实例同时产生日志，支持日志量持续增长

== 改造目标

=== 目标一：业务零影响

审计日志的整个采集、传输、持久化链路对业务请求的性能影响接近于零。业务接口的响应时间不应因审计日志而产生可感知的增加。

=== 目标二：数据零丢失

在任何单点故障场景下（网络抖动、PG 数据库宕机等），审计日志数据不丢失。
系统应具备缓冲和自动恢复能力。

=== 目标三：架构可扩展

系统支持微服务实例数量的横向扩展，支持日志量从当前 10-100万条/天 到未来更高量级的增长，
无需对架构进行重大调整。

== 方案详述

=== 方案一：审计日志直接发送到 Kafka + PG 数据库

==== 方案描述

业务服务中的 `sg-audit-starter` 在采集到审计日志后，直接通过 Kafka Producer 将日志消息发送到 Kafka Topic。
后端的 Consumer 服务（可以是 platform 服务或其他独立服务）消费 Kafka 中的消息，批量写入 PG 数据库。

数据流：**业务服务 → Kafka → Consumer → PG**（3层）

end

=== 方案二：审计日志保存到本地文件 + Vector + PG 数据库（本项目方案）

==== 方案描述

业务服务在采集审计日志时，引入 **Claim Check (大对象卸载)** 模式。超大报文（>=100KB）被异步卸载至 MinIO，本地日志仅记录包含指针的轻量化 JSON。Sidecar 部署的 Vector Agent 负责提取业务元数据并实现物理列入库。

本项目已实现了该方案的完整闭环：

* **Claim Check 卸载机制**：
** **流式处理**：使用 Kryo 流式序列化 `before/after` 对象，消除大内存拷贝。
** **异步压缩**：采用 **GZIP (BEST_SPEED)** 算法，由专用存储线程池异步压缩并上传至 MinIO，对业务主逻辑零干扰。
** **详情指针**：审计详情被替换为 `{"_storage":"MINIO","_url":"..."}` 存储于数据库。
* **元数据物理化 (Metadata Physicalization)**：
** Vector Agent 监听本地日志，动态提取 `business_id`、`trace_id` 等高频检索字段。
** 字段映射：Vector 将这些关键字段直接映射为 PostgreSQL 的 **B-Tree 索引列**，确保秒级精确匹配。
* **懒加载详情还原**：
** 提供详情接口，仅在查看时触发。
** 后端自动识别指针，从 MinIO 下载并执行 **GZIP 解压 -> Kryo 反序列化**，实现数据的透明还原。

数据流：**业务服务(Claim Check) → 本地文件 → Vector → PG**
附件流：**业务服务 → MinIO**

=== 方案三：审计日志保存到本地文件 + Vector + Kafka + PG 数据库

==== 方案描述

在方案二的基础上，引入 Kafka 作为中间缓冲层。业务服务依然执行 **Claim Check** 将大报文卸载至 MinIO，并写入包含指针的轻量化本地日志。Vector Agent 采集并解析日志后，将包含业务元数据的消息发送至 Kafka Topic，由后端 Consumer 批量写入 PG。

该方案适用于极高并发（如单机 TPS > 5000）且对入库实时性要求不高的超大规模场景。

* **优势叠加**：继承了方案二的 Claim Check 优势，Kafka 消息体积极小，吞吐性能极佳。
* **削峰填谷**：Kafka 能够缓冲瞬时极高峰流量，保护数据库在压力波动时不被连接耗尽。
* **链路解耦**：采集层（Vector）与持久化层通过消息中间件彻底解耦。

数据流：**业务服务(Claim Check) → 本地文件 → Vector → Kafka → Consumer → PG**
附件流：**业务服务 → MinIO**

==== 内存增量计算说明

以下详细说明各方案中关键内存增量数据的计算依据。

===== 方案一：业务服务内存增量

方案一在每个业务服务实例中内嵌 Kafka Producer，主要内存消耗来源：

[cols="1,3,2,2"]
|===
| 内存项 | 说明 | 估算值（常规日志） | 估算值（大对象日志）
| Kafka Producer `buffer.memory` | Producer 内部消息缓冲区，用于暂存待发送的消息 | 默认 32MB | 需调大至 **128-256MB**
| Producer 发送线程 | `sender` 线程及相关数据结构 | ~2-5MB | ~2-5MB
| 序列化缓冲区 | 消息 key/value 序列化时的临时缓冲 | ~1-3MB | ~5-15MB
| Metadata 缓存 | Kafka 集群元数据（Topic/Partition/Broker 信息） | ~2-5MB | ~2-5MB
| **单实例合计（常规）** | 审计日志中不包含大型对象快照 | **~37-45MB** |
| **单实例合计（大对象）** | 审计日志中包含大型对象的 oldState/newState | | **~137-281MB**
|===

NOTE: Kafka Producer 的 `buffer.memory` 默认为 32MB，表示 Producer 内部可以缓存的未发送消息总大小。
当审计日志单条达到 100KB 时，32MB 仅能缓冲约 320 条消息，在高峰期极易填满导致阻塞。
因此大对象场景下必须调大 `buffer.memory` 至 128-256MB，否则会频繁触发 `max.block.ms` 阻塞业务线程。


===== 方案二：业务服务内存增量

方案二在每个业务服务实例中仅使用 Logback AsyncAppender，无 Kafka 依赖，主要内存消耗来源：

[cols="1,3,2,2"]
|===
| 内存项 | 说明 | 估算值（常规日志） | 估算值（大对象日志）
| Logback AsyncAppender 队列 | 异步日志队列（当前配置 `queueSize=2048`） | 2048 × 2KB ≈ **4MB** | 2048 × 100KB ≈ **200MB**
| 日志事件对象 | 队列中的 Logback LogEvent 对象开销 | ~2-5MB | ~5-10MB
| JSON 序列化缓冲 | ObjectMapper 序列化时的临时缓冲 | ~1-3MB | ~5-10MB
| RollingFileAppender | 文件输出流及缓冲区 | ~2-5MB | ~2-5MB
| **单实例合计（常规）** | 审计日志中不包含大型对象快照 | **~9-17MB** |
| **单实例合计（大对象）** | 审计日志中包含大型对象的 oldState/newState | | **~212-225MB**
|===

NOTE: 审计日志中包含 `oldState` 和 `newState`（完整的 JSON 对象快照），当被审计的资源对象较大时
（如包含大量字段的配置对象、嵌套集合等），单条日志可能达到 100KB 甚至更大。
此时 AsyncAppender 队列的内存占用将显著增加（`queueSize=2048` 时可达 200MB+）。

===== 方案三：业务服务内存增量

方案三的业务服务部分与方案二完全相同（均通过 Logback 写本地文件），因此单实例内存增量同样取决于日志大小：

* 常规场景：**10-20MB/实例**
* 大对象场景（`queueSize=512`）：**60-75MB/实例**

但方案三需要额外的 Kafka 集群和 Consumer 服务：

* Kafka 集群：复用已有，增量可忽略
* Consumer 服务：~512MB-1GB

|*数据可靠性*| 依赖Kafka | 依赖本地文件+Vector重试 | 依赖本地文件+Kafka双重保障
|*业务服务资源占用*| 高 | 极低 | 极低
|*总新增资源占用*| 中等 | 低 | 中等
|*组件数量*| 业务服务 + Kafka + Consumer + PG | 业务服务 + Vector + PG | 业务服务 + Vector + Kafka + Consumer + PG
|===

== 总结与推荐

=== 推荐方案：方案二（本地文件 + Vector + PG）

=== 推荐理由

==== 1. 核心诉求匹配度最高

方案二完美匹配了前期分析的四项核心诉求：

[arabic]
. **解耦**：业务服务只写本地文件，完全不感知下游存储架构，不依赖任何外部服务（Kafka、platform），日志写入失败不影响业务流程
. **高性能**：业务线程只做一次本地文件写入（通过 Logback AsyncAppender 异步化），对业务接口的响应时间影响在亚毫秒级别
. **高可靠**：本地文件本身就是缓冲区，即使 PG 数据库短暂不可用，日志仍然写入本地文件不会丢失；Vector 恢复后从 checkpoint 继续采集和写入
. **可扩展**：新增微服务实例只需挂载共享日志卷和部署 Vector Agent，无需修改任何配置

==== 2. 架构最简洁

* 仅 4 层链路（业务服务 → 本地文件 → Vector → PG），相比方案三减少了 Kafka 和 Consumer 两个环节
* 组件数量最少（业务服务 + Vector + PG），运维和排错成本低
* 不依赖 Kafka 集群的可用性，减少了外部依赖点

==== 3. 资源利用率最优

* **业务服务资源占用极低**：每个实例仅增加 10-20MB 内存和 <1% CPU（相比方案一）
* **Vector Agent 轻量高效**：Vector 使用 Rust 编写，资源占用极低（50-100MB/实例），以 Docker Sidecar 模式与业务服务共同部署
* **无需 Kafka 集群**：相比方案一和方案三，省去了 Kafka 的资源开销和维护成本
* **无需独立 Consumer 服务**：Vector 直接写 PG，省去了 Consumer 服务的资源开销

==== 4. 改造工作量最小

* `sg-audit-starter` 的改动极小，核心日志采集逻辑保持不变，仅确保日志输出格式和 Logback 配置正确
* 业务服务无需引入新的依赖（Kafka客户端），降低了依赖冲突和版本管理的风险
* 不需要开发和维护独立的 Consumer 服务

==== 5. 当前日志量级完全匹配

当前日志量级为 10-100万条/天，Vector 直接批量写入 PG 数据库完全能够承载。
Vector 内置了批量写入和重试机制，在当前量级下 PG 数据库不会成为瓶颈。
方案三的 Kafka 缓冲层在当前量级下属于过度设计，增加了不必要的系统复杂度。
